# 2026-04-23 旁路通讯 —— 进程间通讯 MVP

## 概述

落地了旁路通讯的第三类资源 —— **进程间通讯（IPC）**。这是继三缓冲（状态同步）、环形队列（事件流）之后的第三类旁路资源，用于跨进程数据搬运。

核心主张：**「进程内静态、进程间动态」**。节点侧代码契约与进程内通讯一致，但进程边界由 URI + Rendezvous 文件动态对接，不穿透 `#[system]` 图规约。

## 与进程内旁路通讯的差异

| 维度 | 进程内 (`triple_buffer` / `ring_buffer`) | 进程间 (`ipc::*`) |
|------|-------------------------------------|------------------|
| 绑定时机 | 编译期 `#[system]` 宏静态确定 | 运行时通过 rendezvous 动态发现 |
| 资源归属 | 系统图节点 | 独立 IPC 资源，运行时注入 |
| 类型安全 | Rust 类型系统 | `SchemaId` 指纹 + 握手双侧校验 |
| 失败模式 | 几乎无 | `NotReady` / `PeerGone` / `SchemaMismatch` |
| 延迟 | 纳秒级（共享内存） | 100 μs（TCP，MVP 后端） |

## 设计决策

### 1. URI 三要素身份

端点通过 `roplat-ipc://<namespace>/<endpoint>?msg=<schema_id>&v=<version>` 唯一标识。

**为何不直接用字符串名**：跨进程不共享 Rust 类型系统，字段错配会引发未定义行为。`schema_id`（编译期计算的结构指纹）与 `msg_version`（手动递增的语义版本）双重校验，在握手阶段就过滤错配流量，拦截在数据层之外。

MVP 阶段 `SchemaId` 由调用方手动声明；后续 `#[roplat_msg]` 宏会自动生成 `const SCHEMA_ID: SchemaId`（见「遗留项」）。

### 2. 文件系统 Rendezvous

发布者写入 `$ROPLAT_RUNTIME_DIR/ipc/<namespace>/<endpoint>.rdv`（JSON），订阅者读取并连接。

**为何不用 DDS / mDNS / etcd**：

- 零外部依赖 —— 嵌入式板卡、离线 CI、容器镜像无差别运行
- 行为可预测 —— 没有多播抖动、没有后台发现线程
- 可诊断 —— 运维直接 `cat *.rdv` 看到所有活跃端点

**僵尸清理**：rendezvous 读取时检查 `publisher_pid` 存活性。Linux 走 `/proc/<pid>`，Windows 用 `OpenProcess` FFI（零额外 crate 依赖）；其他 Unix 平台保守返回 true 由后端连接失败兜底。

### 3. Late Binding 状态机

订阅者连接失败**不是错误**，而是返回 `IpcError::NotReady`。节点侧的 `try_pop()` 行为与「队列为空」完全一致，用户代码形态不因进程边界而改变。

状态机：`Pending → Connecting → Ready`，断连自动回到 `Pending` 并指数退避重连（后续迭代补完自动重试层；MVP 依赖调用方显式重试）。

### 4. Transport trait 解耦

`IpcTransport` 把「搬字节的方式」从语义层（ring_buffer / triple_buffer）抽走。MVP 的 `IpcRingBuffer<T>` 只依赖 trait，不关心底层是 TCP / UDS / SHM。

后端选择由**发布者**通过 rendezvous 描述符公告，订阅者自动适配。切换共享内存后端时，用户代码零改动。

### 5. JSON 握手 + 二进制帧

- 握手（`Hello` / `HelloAck`）：JSON 行协议，一次性开销、优先可读性与可诊断性
- 数据帧：`u32 BE 长度前缀 + payload`，定长解析、无状态

此混合策略常见于工业协议（如 NATS、Kafka），兼顾诊断与性能。

### 6. MVP 后端选择 TCP 而非 UDS

理由：

- **跨平台无歧义**：Windows UDS 需要 1809+ 且无 abstract socket 约定；命名管道又是另一套 API。先用 `std::net::TcpStream` 覆盖全平台。
- **延迟够 MVP 用**：loopback TCP 在本机约 100 μs，远高于进程内但远低于 ROS 2 + DDS 的典型 500 μs–2 ms。
- **后续无痛替换**：`IpcTransport` 的抽象已就位，UDS/SHM 后端只是新实现。

### TCP 后端线程模型

- **发布者**：`TcpListener` 非阻塞 `accept` 循环 → 每个订阅者握手后分出 `reader + writer` 线程对
- **订阅者**：单条 `TcpStream` + `reader + writer` 线程对
- 共享 `Arc<Mutex<VecDeque>>` 做 outbox/inbox，`AtomicBool` shutdown 信号；`Drop` 时清理 rendezvous 文件并 join 所有线程

线程数：发布者 `1 + 2N`（N = 订阅者数），订阅者固定 3（主 + reader + writer）。MVP 可接受；未来共享内存后端将消除 writer 线程。

## 已知限制

1. **容量语义**：MVP 的 `IpcRingWriter::try_push` 背靠 TCP 后端的内存队列，近似无界 —— `try_push` 几乎总返回 `true`。进程内 ring_buffer 的容量语义要等共享内存后端才能真正实现。
2. **`IpcTripleBuffer` 缺席**：三缓冲的「始终最新」语义跨进程需要共享内存 + 双边原子 ready 指针。MVP 只提供 ring_buffer 语义。用户暂时用 `capacity=1` 的 ring_buffer 近似三缓冲 + `force_push` 即可。
3. **多订阅者广播语义**：MVP 的 TCP 后端已支持多订阅者连接到同一发布者，但所有订阅者共享同一个 outbox（先到先得，非广播）。正确的一写多读需要为每个订阅者维护独立 outbox 副本。
4. **`T: Copy` 约束**：字节拷贝假设 T 是 `#[repr(C)] + Copy` 的 POD。`String`、`Vec<T>` 等需等后续序列化器层（bincode / capnp）接入。
5. **无自动重连层**：断连后上层要自己调 `connect_subscriber` 重建；后续要在 `IpcRingReader` 内嵌状态机。
6. **无背压**：TCP 后端的 outbox 内存队列无上限，慢订阅者会拖垮发布者内存。下轮要加 `high_watermark`。

## 下一步计划

按优先级：

1. **`#[roplat_msg]` 宏扩展**：自动生成 `const SCHEMA_ID: &str`、`const SCHEMA_FULL: &str`（完整结构说明）、`const MSG_VERSION: u16`。消除手动维护 SchemaId 的错配风险。
2. **UDS / Named Pipe 后端**：把本机延迟从 100 μs 降到 10 μs。
3. **IPC 两进程 e2e 测试**：在 `examples/` 下新增 `ipc_pubsub/`，`cargo test` 中起子进程验证真实跨进程路径。
4. **`IpcTripleBuffer` + 共享内存后端**：完成三缓冲的跨进程版本；延迟目标 < 1 μs。
5. **`#[system]` 宏识别 IPC 端点**：让系统图可以声明「这条边是进程间」，自动生成 rendezvous 配置。
6. **`#[replay]` 跨进程录制**：IPC 消息写入同一录制流，保持「回放一致性」主张。

## 增量更新（同日第二轮）

在 MVP 基础上继续补完三项：

### 1. TCP 后端真广播（多订阅者独立 outbox）

原先 MVP 版本所有订阅者共享同一个 outbox，导致「先到先得」而非广播。重写为：

- 发布者持有 `Arc<Mutex<Vec<ByteQueue>>>` 的 peer 队列列表
- `accept_loop` 为每个新订阅者分配独立 `ByteQueue` + 独立 writer 线程
- `publish` 时对所有 peer queue 逐一 `push_back(bytes.clone())`
- writer 线程退出时用 `Arc::ptr_eq` 从队列列表中摘除自己，维护 `connected_peers` 计数

引入 `SendSide` 枚举区分 Publisher / Subscriber 的发送策略，避免两种场景共用字段。

### 2. `#[roplat_msg]` 自动生成 SCHEMA_ID / MSG_VERSION

`roplat_macros/src/msg.rs`：

- `MsgAttrArgs` 解析器支持 `version = N` 参数（与 `lang = "..."` 可叠加）
- 新增 `build_schema_full()` 生成可读字段序列，例如 `Pose{seq:u64,x:f32,y:f32,z:f32}`
- 新增 `schema_id_from_full()`：FNV-1a 64-bit 哈希，截取低 48 位为 12 位十六进制（碰撞概率 ~ 2⁻²⁴，MVP 足够）
- 透明模式 `impl` 块注入三个关联常量：`SCHEMA_ID` / `SCHEMA_FULL` / `MSG_VERSION`

双侧编译器从同一源码算出相同 `SCHEMA_ID`，IPC 层握手时直接比对字符串，消除手动维护哈希的错配风险。

### 3. `examples/ipc_pubsub` 示例

新建二元 binary crate：`publisher.rs` + `subscriber.rs` 共享 `msg.rs`。

- 消息声明：`#[roplat_msg(version = 1)] #[repr(C)] struct Pose { seq, x, y, z }`
- 订阅者自动重试 `NotReady` 循环，演示延迟绑定
- README 说明「改动 msg.rs 一侧字段，另一侧不改」会在握手期触发 `SchemaMismatch` 的双保险行为
