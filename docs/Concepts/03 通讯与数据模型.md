# 通讯与数据模型

## 主流通讯与旁路通讯

1. 主流通讯：节点输入输出主链，强类型、可分析。
2. 旁路通讯：用于状态同步等扩展场景。

## 旁路通讯原语

旁路通讯提供两种互补的通讯原语：

### 三缓冲（状态同步）

- **语义**：始终保持最新数据，旧值被覆盖
- **拓扑**：SPMC — 一个 Publisher，多个 Subscriber
- **读端**：`get_latest()` 返回最新值的借用 `Option<&T>`
- **适用场景**：传感器状态、位姿更新等只关心最新值的场景
- **有损性**：发布频率高于读取频率时，中间值会被跳过

### 环形队列（事件流）

- **语义**：FIFO 有序消息传递，不丢失任何已入队消息
- **拓扑**：SPSC — 一个 Writer，一个 Reader
- **读端**：`try_pop()` 返回队首元素的所有权 `Option<T>`
- **适用场景**：指令序列、日志事件、轨迹点等需要有序且不丢失的场景
- **满策略**：
  - `try_push`：满时返回失败，不阻塞（SPSC 安全）
  - `force_push`：满时丢弃最旧元素（不可与 try_pop 并发）

### 如何选择

| 场景 | 推荐原语 |
|------|----------|
| 传感器实时状态 | 三缓冲 |
| 电机命令流 | 环形队列 |
| 位姿广播给多个订阅者 | 三缓冲 |
| 日志/事件/轨迹点 | 环形队列 |
| 只关心最新值 | 三缓冲 |
| 不能丢任何消息 | 环形队列 |

## 透明类型与不透明类型

1. 透明类型：跨语言可见布局（`#[repr(C)]`），字段在 Rust、C++、Python 三端可见。
2. 不透明类型：Rust 侧只保留类型标记，字段由目标语言用户文件定义。

两种旁路通讯原语均支持透明类型和不透明类型作为传输内容。容器（Writer/Reader/Publisher/Subscriber）的 `TypeBinding` 由内部 payload 类型决定。

## 多语言映射

三缓冲和环形队列均遵循三层封装模式：

1. **Rust FFI 内核**：`#[repr(C)]` 控制块 + C ABI 函数（`extern "C"`）
2. **C++ 模板封装**：在 `roplat_cpp/include/roplat/comm.h` 中提供 RAII 模板类
3. **Python ctypes 封装**：在 `roplat_py/src/comm.py` 中提供 ctypes 包装类

各语言通过 `create_triple_buffer` / `create_ring_buffer` 工厂函数创建通道。

## 为什么 opaque 要"空壳化"

核心是避免双重真相：

1. 如果 Rust 和 C++ 同时定义字段，长期容易漂移。
2. 空壳化后，Rust 负责调度，目标语言负责数据实现。

## 工程落地规则

1. 能用透明类型就先用透明类型。
2. 确认是语言私有实现再引入 opaque。
3. 每次引入新数据类型后都跑一遍 lint 与构建验证。
