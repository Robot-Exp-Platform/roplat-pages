# 设计思想

## 设计思想

设计思想是一个项目开始最重要的事情，这是第一性原则的体现，只有在设计思想上达成共识了，后续的开发才有一个明确的方向和目标。我们在设计思想上达成了以下共识：

roplat 是一个希望能够相较于 ros 用动态图规划节点而改用一定的静态图约束的机器人操作系统，得益于编译期静态图已知，其可以获得更多的确定性。故而其希望类似 rust ，成为一款面向运行期安全（即为编译期做更多安全处理）的机器人操作系统。

不同于 ros 和 dora-rs 采用了 节点和节点间通讯的形式来实现整个网络的搭建。roplat 更加注重时间的确定性和输入输出的确定性， 故而采用了输入输出节点（ node，输入输出将作为泛型成为节点特征的一部分) 和时序节律源（rhythm, 类似单片机中的时钟和中断系统，但是我们尝试使用异步而非打断的形式避免更多的上下文切换）, 并使用系统（system）作为封装单元 。出于对于编译期操作的高要求，这使得我们需要采用 rust 语言作为系统内核。

但是并非说 roplat 不能够处理动态图，roplat 维护进程内的静态和进程级的动态，前者是指在编译期就已经确定了节点的输入输出和时序节律源的关系，后者是可以在运行期启用或者弃用某些进展所维护的静态图从而实现动态效果。我们希望通过这样的设计能够在保证系统的确定性的同时也能够满足一些动态的需求。一般来说，一个机器人操作中大多数逻辑依赖都是静态的，动态反而是少见的。这也是我们有底气说我们是对于 ROS 系统的优化的原因了。

某种意义上来说， ROS 的节点组织形式是 Roplat 中所有节点都无关且使用内部旁路数据流通讯的极限特例。
