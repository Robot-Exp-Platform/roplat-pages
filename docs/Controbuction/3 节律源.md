# 3 节律源

节律源决定“何时触发节点执行”。

你可以把它理解成系统的触发器或时钟源。

## Rhythm Trait（当前实现）

```rust
#[async_trait::async_trait]
pub trait Rhythm {
    type Yield;
    type Feed;

    type Output;
    type Error: Into<RoplatError> + Debug;

    async fn drive<N, F>(&mut self, mut nodes: N, op_domain: F) -> Self::Output
    where
        N: Send,
        F: for<'a> FnMut(&'a mut N, Self::Yield) -> BoxFuture<'a, Self::Feed> + Send;
}
```

> **注意**：`drive` 不要求 `N: Sync`。节点元组通过 `&mut` 引用在闭包中逐一解包，不存在并发共享，因此 `Send` 足矣。

`Yield` 是节律源每次触发时交给节点域的数据。

## 内置节律源

1. `SysTimer`

- 固定周期触发
- 产出 `EventMeta`（包含 `timestamp`、`delta`、`jitter`、`sequence`）

1. `CountRhythm`

- 按次数触发
- 可通过 `CountHandle` 动态调整目标次数

1. `EventRhythm<T>`

- 基于事件通道触发
- 由 `EventTrigger<T>` 主动投递事件

## 什么时候选哪一种

1. 控制回路：优先 `SysTimer`
2. 有限步测试：优先 `CountRhythm`
3. 外部事件驱动：优先 `EventRhythm<T>`

## 实践注意

1. 你的 `drive` 循环里不要长期阻塞。
2. 时间敏感场景应关注 `EventMeta.jitter`。
3. 节律触发频率要与节点计算耗时匹配，避免持续积压。
