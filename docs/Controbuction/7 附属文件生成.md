# 7 附属文件生成

由于 `build.rs` 执行早于过程宏展开，构建脚本无法直接读取宏展开结果。因此当前采用“两遍扫描（Two-pass）”方案：

1. 第一遍：提取元数据（Extraction）

- 用户执行 `cargo build`。
- `build.rs` 发现当前不是 EXTRACT 阶段后，启动子进程：
  - `ROPLAT_PHASE=EXTRACT cargo check -p <package_name>`
- 子进程中，过程宏在 EXTRACT 模式下仅做“信息提取”，输出 JSON 清单。

2. 第二遍：读取清单并生成文件（Generation）

- 主进程读取 manifest JSON。
- 根据条目类型生成节点/消息文件。

3. 最终编译与链接（Final）

- `build.rs` 结束后，Cargo 正常继续 Rust 编译与链接。

## 当前清单条目

- `kind = "node"`：来自 `#[roplat::node(lang = ...)]`
- `kind = "msg"`：来自 `#[roplat::roplat_msg]`（透明）
- `kind = "opaque"`：来自 `#[roplat::roplat_msg(lang = ...)]`（不透明）

## 当前文件生成规则

1. 多语言节点

- 生成系统基类：每次覆盖
- 生成用户实现模板：仅首次生成

2. 透明消息

- 每个类型生成一个系统管理文件（默认 C 头文件）

3. 不透明数据

- 每个类型只生成一个用户文件（按 `lang` 落到对应语言目录）

## 默认目录

- 系统管理目录：`cpp/roplat_gen`、`py/roplat_gen`
- 用户目录：`cpp/src`、`py`

以上目录可在 `build.rs` 中通过 `BuildOrchestrator` 改写。

## 为什么采用两遍扫描

- 避免 `build.rs` 与过程宏展开时序冲突
- 宏负责“语义提取”，构建脚本负责“工程落盘”
- 让多语言代码生成具备稳定、可复现、可审计的输入输出
